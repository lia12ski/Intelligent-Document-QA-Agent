# 系统设计说明

## 架构

6 层流水线：PDF 类型判断 → 解析 → 切块 → 索引 → 检索决策 → 生成+自检。每层可单独换，不耦合。

## 为什么这么选

**Parser: Local vs MinerU.** 本地解析免费、离线可用、对文本层 PDF 够用；MinerU 表格解析强但要在线上传轮询，慢且有调用成本。所以做成 provider 可切换，默认 local，遇到复杂表格开 mineru。

**OCR: PaddleOCR vs Tesseract.** PaddleOCR 中文识别率更高，但 pip 安装会拖一堆 paddlepaddle 的依赖。Tesseract 轻量但中文需要额外训练数据。实测样本上 PaddleOCR 效果好得多，所以用它做默认，Tesseract 当备选。

**检索: BM25 + Dense + RRF.** 只用 BM25 的话"财务报表覆盖的期间"和"报告期"这种同义表达匹配不上。只用 Dense 的话短查询（如"金额"）向量精度不够。两个都做再用 RRF 合并排序，实测比单路高 10-15% 召回。Dense 后端用 FAISS 是因为它够快、内存索引零配置，不需要搭服务。如果以后要持久化会换成 Chromadb或者Milvus。

**LLM: DeepSeek（默认）/ OpenAI / Ollama.** DeepSeek 便宜且中文效果好，Ollama 可以本地跑避免数据出去。做成 client 接口隔离，传 prompt 返回字符串，具体用哪个由环境变量配。

**自检独立于 LLM.** 原因很简单：让 LLM 自己判断"我有没有在编"它总说没有。所以检索分数、关键词覆盖、数字一致性这些硬指标单独算，不靠 prompt。

## 边界情况怎么处理的

- **查不到** → 自检不过直接拒答，不硬编
- **打错字**（账面价植）→ 查询阶段 OCR 替换表修正
- **数值题** → 确定性提取直接从 chunk 文本定位合计行取数，不经过 LLM
- **模糊查询**（金额是多少）→ 改写展开具体术语，答案里说明"金额指向不够具体"
- **条款号误判**（5.1 被当成数字查询）→ 数字检测前先剥离条款编号
- **外部依赖挂了**（FAISS 没装、MinerU 超时、LLM 调用失败）→ 每层有降级，不崩

## 验证

55 个单元测试覆盖各模块核心逻辑。6 个端到端用例（正文检索、表格数值、拒答、模糊查询、OCR 容错、跨表综合），每次改完跑一遍确认没坏。三个量化指标：retrieval_hit_rate、citation_rate、self_check_match_rate，当前都是 1.0。

## 遗留问题

- 表格数值提取目前只适配了财报的高管薪酬明细表结构，换一种表格式样就取不出来了。其他场景得上 MinerU 或者版面模型。
- 评估集只有 6 个财务用例和一个标准文件用例，远远不够。多做几个领域、多补几条 case 才能说明系统泛化能力。
- Dense 检索的 embedding 用本地 BGE 小模型，精度和更好的模型有差距，但是免费 + 数据不出网，看场景取舍。
